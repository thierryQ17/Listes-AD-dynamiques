### Piège corrigé — template literal `pageScript` (groups-doc.js)
Le handler de copie de la **page générée** vit dans un **template literal**. Un `/\n+$/`
(ou un `\'`) y devient un **vrai saut de ligne / une vraie apostrophe** → littéral regex /
string cassé → **SyntaxError → tout le script de la page mort → clic `<>` sans effet**.
Correctifs : `.trim()` (sans backslash) et reformulation sans apostrophe. Vérifié en
évaluant le `pageScript` généré exactement comme le navigateur.

## 3. Nouveau cache **majAD** — 1 requête AD, arbre Department → Office

### Besoin
Un cache calculé par **UNE seule requête AD** (au lieu du warmup par site), sur les comptes
fiabilisés par le **traitement paie de 7h** (tag `extensionAttribute15 = majAD`, issu du CSV
`…\MAJ AD\260706-0700_maj_ad.csv`). Arbre **Department → Office → utilisateurs**, même
interface que l'Explorateur.

### Décisions verrouillées (après plusieurs `/analyse`)
- Construction par **requête AD unique** (pas lecture directe du CSV) → objet **identique**
  au cache global → réutilisation totale (Détail, colonnes, tri). Aucun champ manquant.
- Filtre **paramétrable** (`parametres.json → ad.majAdFilter`) : bascule prod = 1 ligne.
- **DC pivot** ciblé : `ad.server = "A31000A00S002"` (via `-Server`, cache majAD uniquement).
- Le fichier `_users_majad.json` est **généré puis lu jusqu'à régénération** (règle du projet
  « tout passe par le cache » ; seule `Build-*Cache` touche l'AD).
- ↻ Cache : **option A** — reconstruit AD **et** majAD.

### Implémentation
- `ad-reader.psm1` : `Build-MajAdUsersCache` (1 `Get-ADUser`, filtre config, `-Server` config,
  projection identique au global) + `Get-MajAdUsersFromCache` + `_users_majad.json`.
- `http-server.psm1` : `GET /api/majad/users` (lit le fichier ; le construit si absent) +
  `POST /api/majad/preload`.
- `explorer.*` : arbre Department→Office construit **en mémoire** (`buildDeptOfficeTree`),
  sélection d'un bureau = **filtrage mémoire, 0 appel AD**. Objet réutilisé pour Détail/colonnes.
- **Validé hors-ligne sur le CSV réel** : 3562 users → **5 départements / 207 bureaux**,
  aucune perte, aucune collision de clé.

### Diagnostic 408 (preuve, pas supposition)
Au 1er build, `/api/majad/preload` renvoyait **408**. Vérifications :
- DC pivot **joignable** (10.254.4.2, LDAP 389 ouvert) ; requête 1-user **rapide**.
- Requête **complète** pivot mesurée : **4416 users en 18 s** → pas trop lent.
- Cause réelle : **timeout de requête Pode** (30 s par défaut) dépassé par build +
  sérialisation. → **`Start-PodeServer -RequestTimeout 600`**.

### Bascule finale : majAD = cache **UNIQUE** de l'Explorateur
- Sélecteur de source retiré ; `state.source='majad'` ; arbre majAD au chargement.
- **Filtre production activé** : `ad.majAdFilter =
  "Enabled -eq $true -and extensionAttribute15 -eq 'majAD'"`.
  ⚠ Tag posé demain 7h → **arbre VIDE d'ici là** (message explicite dans l'arbre).
- **Ancien cache retiré** : plus de warmup par site dans `/api/users/preload`.
  ↻ Cache = **1 requête majAD + 1 requête globale** (global conservé : lu par
  Écarts / Règles / Groupes). `refreshMajAdCache()` côté front.
